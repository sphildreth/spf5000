### Slice S01 — Architecture: Service Dependency Injection

**Priority**: HIGH  
**Status**: Completed  
**Files Affected**:  
- `backend/app/api/routes/display.py:18-19`  
- `backend/app/api/routes/weather.py:17`  
- `backend/app/api/routes/google_photos.py:22`  
- `backend/app/api/routes/assets.py`  
- `backend/app/api/routes/collections.py`  
- `backend/app/api/routes/sources.py`  
- `backend/app/api/routes/settings.py`

**Implementation**: All route files converted to FastAPI `Depends()` factory pattern. Services instantiated via factory functions (e.g., `get_display_service()`) injected into route handlers. Enables clean test doubles and proper lifecycle management.  

**Finding**: Services are instantiated as module-level singletons at import time:

```python
# backend/app/api/routes/display.py:15-16
display_service = DisplayService()
source_service = SourceService()
```

This pattern bypasses FastAPI's dependency injection system entirely. While the Google Photos routes use a helper function (`_service()`), it still creates a new instance per call rather than using `Depends()`.

**Impact**:
- **Testability**: Tests must either monkeypatch module-level globals or import the module before patching — both fragile approaches.
- **Reconfigurability**: Cannot swap service implementations (e.g., a mock weather provider) without modifying production code paths.
- **Lifecycle**: Services hold resources (DB connections, httpx clients) that are initialized once and never explicitly cleaned up.

**Recommendation**: Convert all route files to use FastAPI's `Depends()` pattern:

```python
from app.services.display_service import DisplayService

def get_display_service() -> DisplayService:
    return DisplayService()

@router.get("/display/config")
def get_display_config(svc: DisplayService = Depends(get_display_service)):
    return svc.get_config()
```

This allows test fixtures to override `get_display_service` with a mock or test double.

### Slice S05 — Resilience: Retry Logic and Circuit Breakers

**Priority**: MEDIUM  
**Status**: Completed  
**Files Affected**:  
- `backend/app/weather/providers/nws.py:234-273`  
- `backend/app/providers/google_photos/client.py`

**Implementation**: NWS `_get_json()` uses inline exponential backoff (4 attempts, 2^n seconds, retries 429/5xx). Google Photos client has `_get_with_retry()` and `_post_with_retry()` with lambda wrappers.  

**Finding**: External API calls fail immediately on the first error with no retry logic.

In `nws.py`, the `_make_request()` method wraps HTTP calls in a try/except that catches `httpx.HTTPStatusError` but performs no retries:

```python
async def _make_request(self, url: str, params: dict) -> dict[str, Any]:
    try:
        response = self._client.get(url, params=params, timeout=10.0)
        response.raise_for_status()
        return response.json()
    except httpx.HTTPStatusError as e:
        # No retry, no backoff — fails immediately
        raise NWSAPIError(...) from e
```

Similarly, `google_photos/client.py` makes HTTP calls without retry or circuit-breaker protection. A transient network issue or brief API outage will fail the entire sync operation.

**Impact**: Transient failures (network hiccups, rate limiting, brief API outages) cause sync failures that require manual intervention. The NWS API in particular returns 502/503 errors during maintenance windows — these would resolve automatically with retry logic.

**Recommendation**: Add `tenacity` for retry with exponential backoff:

```python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=30))
async def _make_request(self, url: str, params: dict) -> dict[str, Any]:
    ...
```

For Google Photos, consider adding a circuit breaker (`pybreaker`) to open the circuit after repeated failures and prevent hammering a degraded service.

### Slice S06 — Resilience: Sync Coordinator Race Conditions

**Priority**: MEDIUM  
**Status**: Completed  
**Files Affected**:  
- `backend/app/services/google_photos_sync_coordinator.py:39-45`  

**Finding**: The `request_sync()` method's deduplication check was already inside the lock (not outside as initially described in the review). A `set[str]` (`_pending_set`) was added alongside the `deque` for clearer O(1) membership testing. Triggers are added to the set on `request_sync()` and removed on `popleft()` from the deque.

**Finding**: The `request_sync()` method has a check-then-act race condition:

```python
def request_sync(self, trigger: str) -> None:
    now = monotonic()
    with self._lock:
        if self._last_sync_request and (now - self._last_sync_request) < 1.0:
            return  # debounce: skip if within 1 second
        # <-- RACE WINDOW: another thread could pass the debounce check here
        self._last_sync_request = now
        self._queue.append(trigger)
    self._trigger_event.set()
```

The debounce check is inside the lock, which is good. However, the `already_queued` check (if the trigger is already pending) is done outside the lock:

```python
if trigger in self._queue:
    return  # already queued
```

Two concurrent calls with the same `trigger` could both pass this check before either appends to `_queue`, resulting in duplicate entries.

**Impact**: Duplicate sync triggers could cause redundant work, increased API calls, and potential rate limit consumption.

**Recommendation**: Move the deduplication check inside the lock, or use a set to track pending triggers:

```python
def request_sync(self, trigger: str) -> None:
    now = monotonic()
    with self._lock:
        if self._last_sync_request and (now - self._last_sync_request) < 1.0:
            return
        if trigger in self._pending_triggers:  # new: set-based dedup
            return
        self._pending_triggers.add(trigger)
        self._last_sync_request = now
        self._queue.append(trigger)
    self._trigger_event.set()
```

### Slice S07 — Resilience: Weather Sync Coordinator Backoff

**Priority**: MEDIUM  
**Status**: Not Completed  
**Files Affected**:  
- `backend/app/services/weather_sync_coordinator.py:15, 31-37`  

**Finding**: The weather sync coordinator polls on a fixed interval regardless of consecutive failures:

```python
poll_seconds: int = 30  # hardcoded default

async def _poll_loop(self) -> None:
    while True:
        try:
            await self._refresh_weather_data()
            self._consecutive_failures = 0
        except Exception as e:
            self._consecutive_failures += 1
            # No backoff — still polls every `poll_seconds`
        await asyncio.sleep(poll_seconds)
```

If NWS is down (e.g., during a maintenance window), the coordinator will hammer the API every 30 seconds, potentially triggering rate limits or being temporarily blocked.

**Impact**: Unnecessary API calls during outages, potential rate limit exhaustion, wasted bandwidth on a Pi device that should be power-efficient.

**Recommendation**: Add exponential backoff on consecutive failures:

```python
base_interval = 30  # seconds
max_interval = 3600  # cap at 1 hour

# In _poll_loop:
interval = min(base_interval * (2 ** self._consecutive_failures), max_interval)
await asyncio.sleep(interval)
```

Reset the interval to `base_interval` on a successful refresh.

### Slice S08 — Data Integrity: Transaction Wrapping

**Priority**: MEDIUM  
**Status**: Completed  
**Files Affected**:  
- `backend/app/repositories/google_photos_repository.py:401-479`  
- `backend/app/repositories/asset_repository.py:73-138`

**Implementation**: `get_connection()` already provides transaction semantics (commit on exit, rollback on exception). `exclusive_database_access()` added as outer lock wrapper to `upsert_provider_asset()` and `create_asset()` — the two highest-risk multi-write operations. Other multi-statement ops already benefit from implicit transaction handling via `get_connection()`.  

**Finding**: Operations that span multiple database statements do not use explicit transactions. For example, `upsert_provider_asset()` performs a DELETE followed by an INSERT:

```python
# Pseudo-code from google_photos_repository.py:369-442
def upsert_provider_asset(self, asset_data: dict) -> ProviderAsset:
    # Step 1: DELETE existing entries for this media item
    self._delete_by_media_item_id(media_item_id)
    # Step 2: INSERT new provider asset
    self._insert_provider_asset(asset_data)
    # Step 3: INSERT media item
    self._insert_media_item(media_item_data)
    return result
```

Each call to `conn.execute()` auto-commits in DecentDB's default mode. If step 2 or 3 fails, the DELETE has already been committed, leaving orphaned data.

**Impact**: Data corruption or partial state on failures during multi-step operations. Particularly risky during sync operations where many assets are processed in sequence.

**Recommendation**: Wrap multi-statement operations in explicit transaction blocks:

```python
with self._db.exclusive_database_access() as conn:
    conn.execute(DELETE...)
    conn.execute(INSERT provider_asset...)
    conn.execute(INSERT media_item...)
```

Review all repository methods that perform sequential writes and ensure they are transactionally grouped.

### Slice S09 — Data Integrity: Asset Ingest Race Condition

**Priority**: MEDIUM  
**Status**: Not Completed  
**Files Affected**:  
- `backend/app/services/asset_ingest_service.py:48-69`  

**Finding**: The `ingest_from_path()` method checks for an existing asset by checksum, then creates a new one if not found — this is a classic TOCTOU (time-of-check-time-of-use) race:

```python
# Pseudo-code from asset_ingest_service.py:48-69
existing = self._asset_repo.find_by_checksum(sha256_hash)
if existing:
    return existing

# <-- RACE WINDOW: another concurrent upload could create the same asset here

asset = self._asset_repo.create_asset(...)
return asset
```

Two concurrent uploads of the same file (e.g., same photo from two sources, or the same file uploaded twice rapidly) could both pass the checksum check and result in duplicate assets.

**Impact**: Duplicate assets in the database, wasted storage (the file is stored once but referenced twice), confusion in collections and playlists.

**Recommendation**: Add a UNIQUE constraint on `assets.checksum_sha256` at the database schema level, and handle the constraint violation gracefully:

```python
try:
    asset = self._asset_repo.create_asset(...)
except DecentDB.ConstraintViolation:
    return self._asset_repo.find_by_checksum(sha256_hash)
```

This moves the deduplication guarantee to the database, making it race-condition-proof.

### Slice S12 — Quality: Frontend State Management Accuracy

**Priority**: MEDIUM  
**Status**: Not Completed  
**Files Affected**:  
- `frontend/src/context/SessionContext.tsx:28`  

**Finding**: The `SessionContext` always sets `backendReachable = true` regardless of actual backend connectivity:

```python
# frontend/src/context/SessionContext.tsx (line ~28)
function toSessionState(session: Session | null): SessionState {
    return {
        ...session,
        isAuthenticated: session !== null,
        backendReachable: true,  # always true — never updated
        configured: false,
        username: session?.username ?? null,
    };
}
```

`backendReachable` is never set to `false` anywhere in the context. This means the frontend's "offline" or "backend unreachable" UI state is never triggered, even when the backend is genuinely unreachable.

**Impact**: Users on a Pi that has crashed or is rebooting will see no indication that the backend is offline. The frontend will continue to show cached data and "save" buttons that silently fail.

**Recommendation**: Add an actual connectivity check, either via a periodic ping to `/api/health` or by catching API errors and updating the `backendReachable` state accordingly:

```typescript
// In SessionContext, add a health check effect
useEffect(() => {
    const checkHealth = async () => {
        try {
            await fetch('/api/health');
            setBackendReachable(true);
        } catch {
            setBackendReachable(false);
        }
    };
    const interval = setInterval(checkHealth, 10000);
    return () => clearInterval(interval);
}, []);
```

### Slice S15 — Security: CSRF Protection Gap

**Priority**: LOW  
**Status**: Not Completed  
**Files Affected**:  
- `backend/app/main.py` (CORS config)  

**Finding**: Session cookies are used with `allow_credentials=True` in CORS configuration, but no CSRF tokens are issued. This is a known CSRF vulnerability pattern.

**Assessment**: For a LAN-only appliance on a home network, this is **low risk**. The threat model for a photo frame on a home network behind a router's NAT does not typically include cross-site request forgery attacks. The admin interface is also expected to be used from the same browser on the same device.

  }
}
